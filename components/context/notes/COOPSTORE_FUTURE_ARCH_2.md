# Техническое задание на модульную платформу NestJS + Vue (SPA + микрофронтенды)

## 1. Общие требования и архитектура

Модульная платформа должна быть основана на **NestJS** (бэкенд) и **Vue.js** (фронтенд SPA) с поддержкой микрофронтендов. Платформа разворачивается в Docker-контейнере и включает три типа модулей:

- **Core-модули (встроенные публичные)** – базовый функционал, поставляемый вместе с ядром.
    
- **Приватные расширения (internal extensions)** – дополнительные модули, поставляемые как приватные npm-пакеты из приватного репозитория (например, Nexus). Их можно устанавливать без пересборки Docker-образа, активировать/деактивировать и перезагружать приложение для подключения.
    
- **Внешние приложения (external apps)** – отдельные веб-приложения, подключаемые по URL и изолированные через iframe.
    

Архитектура должна обеспечивать **горизонтальную масштабируемость** и **раздельное обновление модулей**. Основная логика разделена по модулям: ядро предоставляет сервисы и API, приватные модули расширяют функционал через заранее определённые интерфейсы (см. раздел 4), а внешние приложения интегрируются поверх SPA через iframe. Для взаимодействия между компонентами используется единый графовый API на базе **GraphQL**, где каждый модуль может поставлять свои резолверы.

**Схема взаимодействия компонентов:** ядро NestJS выступает GraphQL-сервером, в который подключаются резолверы из встроенных и приватных модулей. Фронтенд SPA на Vue загружается контейнерным приложением, которое динамически регистрирует маршруты и состояния внутренних модулей. Внешние приложения открываются в <iframe> контейнере. Взаимодействие между компонентами идёт через GraphQL (данные) и через события/стор (глобальный Vuex или аналог) для коммуникации фронтендов.

_Рис.1. Примерная схема архитектуры модульной платформы: центральное ядро (NestJS+GraphQL), внутренние расширения (кодом и резолверами), внешние приложения (iframe) и общий фронтенд-контейнер (Vue SPA)_.

Основные технологические решения:

- **NestJS** с динамическим подключением модулей. Используется механизм динамических модулей и сервис `ModuleRef` для загрузки приватных расширений во время запуска.
    
- **GraphQL (Apollo)**: один обобщённый сервер, совмещающий схемы/резолверы из всех модулей. Внедряется схема-first или code-first подход. _Важно:_ GraphQL-`schema` является **статическим контрактом** между клиентом и сервером; его динамическое изменение в рантайме нежелательно. Поэтому после установки новых модулей требуется перезапуск сервера (иначе схема не обновится). Альтернативно можно рассмотреть Apollo Federation, но для большинства случаев достаточно перезапуска.
    
- **Vue.js SPA + микрофронтенды:** основное приложение (AppShell) на Vue 3+Vite/Wepack, куда **динамически регистрируются маршруты и состояния** внутренних модулей. Интеграция микрофронтендов осуществляется через загрузку их бандлов по `<script>` (или Web Components) в рантайме. Это позволяет развёртывать каждый фронтенд-модуль независимо и обмениваться данными между ними без iframe. Внешние приложения подключаются отдельно через iframe, что даёт надёжную изоляцию.
    
- **Layer «platform-interfaces» (ранее «contract layer»):** общий npm-пакет с набором TypeScript-интерфейсов и типизаций, используемых ядром и всеми расширениями для взаимодействия (например, определение общих DTO, GraphQL Input/Object типов). Этот публичный пакет должен выпускаться в npm (или храниться отдельно) и служить «контрактом» между ядром и модулями.
    

## 2. Модули платформы

1. **Core-модули (public core):**
    
    - Распространяются вместе с ядром платформы (включаются в основной репозиторий/Docker-образ).
        
    - Содержат базовую бизнес-логику, общие сущности, начальные resolvers и UI-компоненты (Vue).
        
    - Пример: модуль аутентификации, настройки пользователей, базовый дашборд и т.д.
        
    - Разрабатываются и обслуживаются как часть основного кода, их обновление требует пересборки ядра.
        
2. **Приватные расширения (internal extensions):**
    
    - Каждый расположен в **отдельном репозитории** (доступном только для организации) и выпускается как приватный npm-пакет.
        
    - Содержит backend-часть (NestJS-модуль, сервисы, резолверы GraphQL) и, опционально, frontend-часть (Vue-компоненты/маршруты).
        
    - Разработчик расширения внедряет себя в **platform-interfaces**, реализуя нужные интерфейсы (например, подписываясь на события ядра или расширяя API).
        
    - Установка: `npm install <private-package>` в директорию приложения (или настройка автоподключения через Nexus). После установки модуль подключается при перезапуске контейнера (см. раздел CI/CD). Модуль можно активировать/деактивировать через административный интерфейс или конфиг.
        
    - _Примечание:_ хотя NestJS поддерживает «динамическое» подключение модулей, GraphQL-схема статична, поэтому при появлении новых схем потребуется перезапуск сервера или обновление схемы вручную.
        
3. **Внешние приложения (external apps):**
    
    - Независимые веб-приложения (любой технологии), которые отображаются внутри платформы через iframe.
        
    - Передаются пользователю как отдельный URL. Например, панели управления партнёрами, служебные сервисы и т.п.
        
    - Ядро платформы обеспечивает доступ по URL и авторизацию, но функционал внутри iframe изолирован от основной SPA (общение только через межфреймовое взаимодействие или внешние API).
        

_Рис.2. Микрофронтенд-архитектура: основной SPA-контейнер (AppShell) загружает внутренние модули через скрипты, а внешние – в iframe._

## 3. Бэкенд: NestJS + GraphQL

- **NestJS-модули:** Каждый модуль (core или расширение) оформляется как модуль NestJS с контроллерами/сервисами и провайдерами. Приватные расширения подключаются как динамические модули. Например, используя конструкцию:
    
    @Module({})   
    export class ExtensionModule {  
      static register(): DynamicModule {  
        return {  
          module: ExtensionModule,  
          providers: [/* ... */],  
          exports: [/* ... */],  
        };  
      }  
    }
    
    Главный `AppModule` при старте импортирует `ExtensionModule.register()` для всех установленных расширений.
    
- **GraphQL:** В корневом модуле настраивается `GraphQLModule.forRoot<ApolloDriverConfig>({ driver: ApolloDriver, autoSchemaFile: true })`. Все резолверы из импортированных модулей будут автоматически включены в схему. _Важно:_ GraphQL-схема **не поддерживает изменения на лету** – она определяется при старте сервера. Из-за этого для подключения нового резолвера потребуется рестарт приложения. В документации NestJS это учитывается: схема генерируется по итоговому набору @Resolver’ов при инициализации сервера.
    
- **Динамические модули и плагины:** Для загрузки приватных расширений можно использовать механизм **NestJS DynamicModule** и даже Node.js-модуль `vm` (как показано в примере модели плагина). При необходимости можно сделать сервис загрузки плагина, получающего код из приватного репозитория (аналогично [48] с `vm.Script`), или проще — устанавливать пакеты `npm` и импортировать их модуль через конфиг.
    
- **Доступ к платформ-интерфейсам:** Модули взаимодействуют через _interfaces_ из общего пакета `platform-interfaces`. Например, для получения данных ядра модуль может инжектировать сервис, реализующий интерфейс из `platform-interfaces`, или подписываться на события (через событийный эмиттер или Pub/Sub).
    
- **Авторизация и лицензирование:** Все запросы GraphQL проходят сквозную авторизацию. Для приватных модулей перед запуском проверяется наличие лицензии: можно реализовать сервис, который при инициализации модуля запрашивает лицензионный ключ (например, из переменных окружения или стороннего лицензионного сервера) и отключает модуль при несоответствии. Поскольку Docker-контейнер может быть скомпрометирован (см. [20]), логика проверки лицензий должна выполняться на уровне приложения, а не полагаться на защиту контейнера.
    
- **Роль платформ-интерфейсов:** Пакет `platform-interfaces` содержит контракты (например, TypeScript-интерфейсы данных, GraphQL-инпуты/тайпы, события), которых должны придерживаться все модули. Это снижает жёсткую связанность: ядро знает только об интерфейсах, а конкретные реализации приходят из модулей. Такой подход аналогичен паттерну «Ports and Adapters» (где `platform-interfaces` – это порты).
    

## 4. Фронтенд: Vue.js SPA + микрофронтенды

- **SPA-контейнер (AppShell):** Основное приложение на Vue 3 собирается, например, с помощью Vite/Webpack Module Federation или SystemJS. Контейнер содержит общий UI (header, меню, рутовые страницы) и точку монтирования для микрофронтендов.
    
- **Динамическая регистрация маршрутов:** При загрузке в контейнере регистрируются маршруты как для базовых страниц ядра, так и динамически для каждого подключённого модуля. Например:
    
    // Vue Router (динамическая конфигурация)  
    const routes = [  
      { path: '/', component: MainPage },  
      // динамически добавляем:   
      { path: '/ext-module', component: () => import('ext-module/App.vue') },  
      // ...  
    ];  
    const router = createRouter({ routes });
    
    Как указано в рекомендациях, _«router config should handle both static core routes and dynamic micro-frontend paths»_.
    
- **Микрофронтенды (internal):** Внутренние модули представляют собой пакеты, экспортирующие Vue-компоненты и маршруты. При установке нового модуля фронтенд может использовать webpack Module Federation или динамический `import()` по URL модуля. Пример метода: при переходе по маршруту контейнер загружает бандл модуля через `<script>` или `import()`, затем вызывает его entry-point (как описано Фаулером). Это позволяет **развёртывать каждый фронтенд независимо** и обмениваться данными через глобальное состояние или события.
    
- **Vuex и разделение состояния:** Для каждого микрофронтенда рекомендуется использовать _namespaced_ модули Vuex (или Pinia) – отдельный store-namespace для каждого приложения. Это изолирует состояние, но при необходимости можно создать общий сервис/стор для обмена данными. Как правило, каждая команда отвечает за свой slice состояния, что упрощает независимую разработку.
    
- **Внешние приложения (iframe):** Они не интегрируются в SPA напрямую – для них задаётся отдельный путь, открывающий iframe. Например:
    
    <iframe :src="externalAppUrl" style="width:100%;height:100%;"></iframe>
    
    Это обеспечит полную изоляцию стилей и переменных. Однако маршрутизация внутри iframe осуществляется самим приложением в iframe, а не основным роутером.
    

## 5. Репозитории и публикация пакетов

- **Платформа (ядро) и интерфейсы:** Основной код платформы и `platform-interfaces` можно хранить в публичном или внутреннем монорепозитории. Пакет `platform-interfaces` публикуется в npm (public), чтобы и ядро, и сторонние расширения могли зависеть от одного версии контрактов.
    
- **Приватные модули:** Каждый приватный модуль хранится в своём приватном Git-репозитории. В нем настраивается `package.json` с приватным namespace (например, `@org/module-name`) и секретным доступом. При коммите релизной версии сборки npm-пакета CI/CD выкладывает его в частный npm-репозиторий (Nexus/Artifactory) с помощью CLI (токены доступа).
    
- **CI/CD-пайплайны:**
    
    1. **CI для ядра и интерфейсов:** сборка, тестирование (unit, e2e), статический анализ. Публикация пакета `platform-interfaces` на npm (если публично), а ядра – либо как Docker-образ, либо как npm-пакет (если планируется reuse).
        
    2. **CI для приватных модулей:** сборка и тестирование, затем публикация npm-пакета в приватный репозиторий. Используются безлогин-скрипты или токены (см. рекомендации npm для CI/CD). При публикации в Docker-образах ядра модулей нет – они подтягиваются при запуске приложения.
        
    3. **CD для платформы:** при мёрже в main ветку запускается билд Docker-образа приложения (NestJS+Vue). Из образа вынимается фронтенд-ассеты и API. Тэги версий, деплой в Kubernetes/VM.
        
- **Проверка доступа:** CI собирает Docker-образ без приватных модулей. При деплое или после него в среде (DevOps) производится `npm install @org/priv-module --registry=https://nexus/...` с правами доступа. После установки система может перезапустить приложение для подключения модулей.
    

## 6. Лицензирование и безопасность

- **Модель лицензирования:** Для управления доступом к приватным модулям реализуется лицензирующая подсистема. Каждый модуль может иметь свой лицензионный ключ/сертификат, хранящийся в конфигурации (например, в базе данных или защищённом хранилище). При инициализации модуля (или периодически) приложение проверяет валидность ключа: если лицензия просрочена или отсутствует, модуль отключается.
    
- **Безопасность Docker:** Как отмечено в исследованиях, Docker не защищает исходный код: любой, у кого есть доступ к контейнеру, может извлечь файлы. Поэтому критическая логика лицензирования **должна быть зашифрована или обфусцирована**, а важные данные (ключи) – на стороне сервера. Можно рассмотреть отдельный лицензионный сервис (собственный микросервис), откуда приложение при старте запрашивает статус лицензии.
    

## 7. Масштабируемость и производительность

- **Горизонтальное масштабирование:** Платформа должна поддерживать запуск нескольких экземпляров приложения в Kubernetes или Docker Swarm. Для этого сервисы ядра должны быть stateless (кроме обращения к БД/кэшу). GraphQL-сервер масштабируется как любое API.
    
- **База данных и кеширование:** Если модули используют базу, рекомендуется общая БД с неймспейсом (широкая схема) или несколько БД по модулям. Кэширование (Redis) для часто обращаемых запросов.
    
- **Разделение нагрузки:** Можно вынести некоторые публичные модули в отдельные микросервисы (при помощи NestJS Microservices или Federation), но это усложняет архитектуру. В большинстве случаев предпочтительно держать единый сервер и перезапускать при добавлении модулей.
    
- **Vue SPA:** Фронтенд-контейнер лёгок, остальные микрофронтенды загружаются лениво. Использовать tree-shaking и lazy-loading для снижения времени первых загрузок.
    

## 8. Обоснование архитектурных решений

- **GraphQL – статический API-контракт:** Как подчёркивается в сообществе, GraphQL-схема – это контракт между клиентом и сервером, и она **не должна часто меняться**. Изменение схемы «на лету» ведёт к непредсказуемому поведению, необходимости перезапуска сервера и рассинхрону между клиентом и сервером. Поэтому в данной платформе добавление новых типов/резолверов требует рестарта приложения. Альтернативный подход – использовать Apollo Federation или GraphQL Mesh, но это усложняет конфигурацию. На практике достаточно рестартовать сервер при установке новых модулей.
    
- **Динамическая загрузка модулей:** NestJS поддерживает **динамические модули** (DynamicModule) и, при необходимости, загрузку кода из внешних источников (как показано в [48]). Это позволяет реализовать плагиновую архитектуру: ядро загружает модули по URL или из node_modules, предоставляя им заранее определённые провайдеры (через фабрики). Такой подход обоснован тем, что модули могут поставляться независимо, а ядро остаётся неизменным.
    
- **Микрофронтенды vs iframes:** Для внутренних модулей выбран подход **скриптовой интеграции** (или Web Components) без iframe, как рекомендует Фаулер. Это обеспечивает высокую гибкость и обмен данными между микрофронтендами (общий стор, shared events), а также независимый релиз каждого модуля. В то же время, для внешних приложений используется iframe, чтобы полностью изолировать их контекст (стили, переменные) и не пускать посторонний код во внутренний SPA. Такая стратегия сочетает изоляцию (для внешних) и гибкость интеграции (для внутренних).
    
- **Платформа-интерфейсы как «контракт»:** Вместо термина «contract layer» введено понятие `platform-interfaces`. Это публичный пакет интерфейсов/типизаций, выполняющий роль общего API. Такой подход гарантирует, что ядро и расширения «говорят на одном языке», но при этом разделены независимыми репозиториями и релизами. Это усиливает слабую связанность (loose coupling) и упрощает обновление контрактов.
    
- **CI/CD и управляемость версий:** Каждый компонент системы (ядро, интерфейсы, расширение) имеет свою цепочку сборки и публикации. Версионирование пакетов (семантически или тегами) позволяет следить за зависимостями и откатами. При выпуске нового расширения не требуется пересборка всей платформы – достаточно опубликовать пакет и обновить его версию в конфиге приложения.
    

## 9. Заключение

В результате предлагается **модульная, расширяемая система** на NestJS и Vue, удовлетворяющая всем требованиям. Центральная часть – ядро с GraphQL и SPA-контейнер – остаётся легковесной, а функциональность добавляют внешние модули. Архитектура учитывает статичность GraphQL-схемы (требуется перезапуск при расширении), изоляцию внешних приложений (iframe) и гибкую маршрутизацию внутренних (динамические маршруты). Компоненты взаимодействуют через чётко определённые интерфейсы, обеспечивая масштабируемость и безопасность системы.
